# Foundation — el piso analítico

> **Para qué existe:** validar o descartar una hipótesis antes de comprometer presupuesto.

## Cuándo entra

- Hay **una** pregunta de negocio nítida.
- Existe una hipótesis previa (interna del cliente, del equipo de planning, de un workshop).
- La decisión está a 1-3 meses de tener que tomarse.
- El cliente quiere evidencia direccional **antes** de invertir en proyecto grande.

Foundation NO es un proyecto pequeño barato. Es un proyecto **de diagnóstico** con alcance definido — su valor está en filtrar qué problemas merecen Intelligence o Strategy.

## Composición típica

| Atributo                      | Valor                                                      |
| ----------------------------- | ---------------------------------------------------------- |
| Pregunta de negocio           | 1 (única, nítida)                                          |
| Metodologías aplicadas        | 1-2                                                        |
| Tiempo                        | 4-6 semanas                                                |
| Equipo Noisia                 | 2 personas (1 lead + 1 analista)                          |
| Modalidad                     | Proyecto puntual con kickoff, midpoint, entrega            |
| Output principal              | Diagnóstico trazable                                       |
| Modalidad de cobro            | Fee fijo del proyecto                                      |

## Qué se entrega siempre

1. **Diagnóstico de la pregunta** — sí / no / "depende de" con evidencia.
2. **Corpus codificado** — el archivo de evidencia con tags y fuente, propiedad del cliente.
3. **Recomendación de siguiente paso** — si la pregunta abre un proyecto más grande, lo que tendría sentido sería X.
4. **Limitaciones explícitas** — qué Foundation no respondió.

## Qué NO se entrega en Foundation

- Playbook de activación o roadmap de implementación (eso es Intelligence).
- Monitoreo continuo (eso es Strategy).
- Las 6 metodologías combinadas (eso es Intelligence o Strategy).
- Comparativos exhaustivos contra >2 competidores.

## Estructura típica del entregable

```
1. Pregunta de negocio (verbatim del cliente)
2. Hipótesis evaluada
3. Metodología(s) aplicada(s) — qué y por qué
4. Corpus — fuentes, window, tamaño, exclusiones
5. Hallazgos principales (3-7) con confianza calibrada
6. Diagnóstico (sí/no/depende)
7. Limitaciones de esta corrida
8. Próximo paso recomendado
```

20-40 páginas máximo. Si pasa de 40, no es Foundation — es Intelligence vendido como Foundation.

## Cómo se calcula el precio

Ver `02-services/pricing-logic.md`. Variables principales:
- Cuántas fuentes hay que orquestar (más nicho = más caro).
- Window temporal del corpus.
- Mercados (1 país vs. 2-3).
- Comparativo (sin comparativo / 1 competidor / 2 competidores).

No hay rate card publicado.

## Patrones de cliente para Foundation

**Buen fit:**
- Director de marketing/innovación con presupuesto restringido para Q+1, necesita defender la inversión.
- Equipo de planning de agencia que quiere brief defensible antes de proponer campaña.
- Founder de scaleup evaluando si entrar a un mercado nuevo.
- VP de producto evaluando si una feature merece desarrollo.

**Mal fit:**
- Cliente que quiere "panorama de la categoría" sin pregunta específica.
- Procurement que pide "1 proyecto barato para probar".
- Quien busca confirmación de hipótesis ya decidida internamente. Foundation puede confirmar o desafiar — si solo se acepta confirmar, hay riesgo de mala relación.

## Riesgos de Foundation mal-vendido

- Vender Foundation cuando el problema requiere Intelligence: el output va a ser insuficiente y el cliente queda con sensación de que "Noisia no entrega".
- Vender Foundation cuando el cliente espera Strategy: sin compromiso continuo, después no hay quien aplique los hallazgos.
- Inflar el alcance de Foundation a precios de Intelligence: rompe la lógica del tier y la integridad de pricing.

## Transición a Intelligence

Foundation se cierra. Si el diagnóstico abre la pregunta a "ahora necesitamos un playbook accionable", se propone Intelligence como proyecto separado — no como upsell automático. La integridad del cierre es importante: Foundation tiene final.
